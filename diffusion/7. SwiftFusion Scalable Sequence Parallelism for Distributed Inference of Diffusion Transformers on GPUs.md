## 1. 阅读计划回顾

这篇论文不要先读成“又一个 diffusion 模型加速技巧”。更好的入口是：

```text
高分辨率图像 / 长视频 DiT 推理
  -> activation 随 sequence length 变大
  -> 单卡放不下或延迟太高
  -> sequence parallelism 把序列维切到多卡
  -> 但 attention 需要所有 Q 与所有 KV 交互
  -> 所以瓶颈从算力转移到跨 GPU 通信
  -> SwiftFusion 重新安排通信发生在哪里、何时发生、怎么发生
```

建议按以下顺序阅读：

1. `Abstract` 和 `Introduction`：先确认论文目标是 DiT serving engine，不是改变扩散采样质量。
2. `Background`：精读 Ring Attention、Ulysses Attention、USP 的通信模式和通信量。
3. `Key Challenges`：理解三个系统瓶颈：拓扑不匹配、all-to-all 不能 overlap、two-sided communication 同步开销。
4. `SwiftFusion`：按三个贡献读：topology-aware sequence parallelism、Torus Attention、one-sided communication。
5. `Algorithm 1` 和 Appendix C：只在需要实现或检查正确性时精读，重点是 attention output 如何像 FlashAttention 一样增量合并。
6. `Evaluation` 和 Appendix B：不要只抄 speedup，要看每个实验分别证明哪个设计选择有效。

第一遍可以略读 `Related Works`。Appendix D 的通信量证明可以先读结论，等公式阻塞理解时再回头。

## 2. 阅读主线

显式问题是：

```text
当 DiT 生成更高分辨率图像或更长视频时，
如何在多 GPU / 多机器上降低分布式推理延迟并避免 activation OOM。
```

真正的结构张力是：

```text
DiT 的模型权重相对不一定巨大，
但 attention activation 会随 sequence length 线性增长。

把 sequence 切到多 GPU 后：
  每张卡只保存部分 Q/K/V activation
  但每个 Q 仍需要看到全局 KV 才能算完整 attention

因此分布式推理的瓶颈不是“有没有更多 GPU”，
而是“跨 GPU 通信是否比省下来的计算更贵”。
```

第一原则是：

```text
分布式 attention 的性能由三件事共同决定：
1. 慢网络上的通信量有多大；
2. 通信能不能被 attention computation 覆盖；
3. 通信原语是否引入额外同步和 GPU kernel 开销。
```

SwiftFusion 的核心重构是：

```text
不要只问 Ring / Ulysses 哪个更好。

应该问：
  哪一种通信模式适合慢的 inter-machine network？
  哪一种通信模式适合快的 intra-machine interconnect？
  剩下的慢通信能不能拆块并隐藏在计算里？
  通信实现能不能减少 sender-receiver 同步？
```

## 3. 论文要解决的问题

Diffusion Transformer 生成图像或视频时，会经过多个 sampling steps。每一步都要执行一次 DiT，把 noisy latent 逐渐 denoise。随着分辨率和视频时长增长，token / patch / frame 对应的 sequence length 变长，attention activation 迅速膨胀。

论文给出的典型现象是：

```text
CogVideoX 5B:
  model weights 约 12 GiB
  但生成 10s 768 x 1360 视频时，
  activation 会让单张 A100 40GB OOM
```

所以 DiT serving 必须依赖 sequence parallelism：

```text
Q, K, V 沿 sequence dimension 切分到多张 GPU
  -> 每张 GPU 只持有 L / P 长度的 sequence slice
  -> 降低单卡 activation memory
```

但 attention 的本质是：

```text
每个 local Q
  必须和全局所有 K/V 计算 attention
```

因此 sequence parallelism 无法消除通信，只能选择通信模式。已有方法的矛盾是：

```text
Ring Attention:
  通信可以和计算 overlap
  但每 GPU 通信量不随 GPU 数增加而下降

Ulysses Attention:
  通信量随 GPU 数增加而下降
  但 all-to-all 通常不可 overlap
  且并行度受 attention heads 数限制

USP:
  组合 Ring + Ulysses
  但把 Ring 用在 inter-machine network 上
  导致跨机器通信量仍然高
```

SwiftFusion 的判断是：现代 GPU 集群中，机器内 NVSwitch / NVLink 很快，机器间 EFA / InfiniBand 相对慢。因此设计应该优先减少跨机器通信量，而不是让直觉上的“简单 ring pattern”跑在慢链路上。

## 4. 核心符号

```text
B: batch size。
L: 全局 sequence length。
H: attention heads 数量。
D: 每个 attention head 的 hidden dimension。

Q, K, V: attention layer 的 query / key / value tensors。
O: attention output tensor。

P: sequence parallelism 使用的 GPU 数。
N: GPU machines 数量。
M: 每台机器内 GPU 数量。

P_u: Ulysses Attention 的并行度。
P_r: Ring Attention 的并行度。

T, U, R: Algorithm 1 中 Torus / Ulysses / Ring process group 的并行度。
t, u, r: 当前 GPU 在 Torus / Ulysses / Ring process group 中的 local rank。

SFU: SwiftFusion。
TAS: 只使用 topology-aware sequence parallelism 的 SwiftFusion variant。
USP: Unified Sequence Parallelism，论文主要对比基线。

ScatterPush: 用 one-sided put 类原语把本地数据 push 到远端 GPU。
GatherPull: 用 one-sided get 类原语从远端 GPU pull 数据。
Barrier / BarrierAll: 显式同步，保证远端数据一致性。
```

注意 tensor shape：

```text
原始 Q/K/V:
  [B, L, H, D]

P 张 GPU 做 sequence sharding 后，每张 GPU:
  [B, L / P, H, D]

Ulysses all-to-all 后，每张 GPU聚合 full sequence、切分 head:
  [B, L, H / P, D]
```

## 5. 方法框架

SwiftFusion 可以压缩成一句话：

```text
把低通信量的 Ulysses 用在慢的跨机器网络上，
把高通信量但可 overlap 的 Ring 用在快的机器内网络上，
再用 Torus Attention 和 one-sided communication 隐藏 all-to-all 与同步开销。
```

整体结构是：

```text
DiT attention layer
  -> sequence-sharded Q/K/V
  -> topology-aware process mesh
  -> inter-machine: Ulysses / Torus Attention
  -> intra-machine: Ring Attention
  -> one-sided communication implementation
  -> return same attention result with lower latency
```

### 5.1 为什么要反过来用 Ulysses 和 Ring

USP 的直觉是：

```text
机器内网络快:
  用复杂 all-to-all 的 Ulysses

机器间网络慢:
  用 pattern 简单、可 overlap 的 Ring
```

SwiftFusion 认为这个映射漏掉了通信量这一维。Ring 的问题是每 GPU 通信量近似不随 GPU 数下降：

```text
Ring:
  communication volume per GPU ≈ 2BLHD
```

Ulysses 的通信量随并行度下降：

```text
Ulysses:
  communication volume per GPU ≈ 4BLHD / P
```

因此跨机器网络虽然更适合简单通信 pattern，但它更需要低通信量。SwiftFusion 选择：

```text
inter-machine network:
  用 Ulysses 降低通信量

intra-machine network:
  用 Ring 承担较高通信量
  因为 NVSwitch / NVLink 带宽更高、延迟更低
```

这就是 topology-aware sequence parallelism 的核心。

### 5.2 Torus Attention 解决什么

把 Ulysses 放到跨机器网络后，会暴露另一个问题：

```text
Ulysses 的 all-to-all:
  通信量低
  但通常作为原子通信操作执行
  不能和 attention computation overlap
```

Torus Attention 的观察是：

```text
all-to-all 前后，有一部分 tensor chunk 留在当前 rank，
这些 stationary chunks 不需要等通信完成就能开始计算。
```

因此它把 all-to-all 拆成多个 stages：

```text
先计算当前 rank 已经拥有的 chunk
同时发送 / 接收下一批 chunk
收到新 chunk 后立刻计算
同时传输再下一批 chunk
```

这个策略把跨机器 all-to-all 从“先通信、再计算”改成：

```text
compute chunk 0
  overlaps communicate chunk 1
compute chunk 1
  overlaps communicate chunk 2
compute chunk 2
  overlaps communicate chunk 3
...
```

### 5.3 为什么需要 one-sided communication

Ring、Ulysses、Torus 都需要跨 GPU 数据移动。传统 NCCL 这类 two-sided communication 通常要求 sender 和 receiver 配合，容易引入隐式同步。

论文指出两个开销：

```text
1. sender / receiver 需要确认对方 buffer ready
   -> 每一步都可能形成同步 barrier

2. 部分通信由 CUDA kernels 实现
   -> 占用 GPU SM
   -> 和模型计算 kernel 竞争
```

SwiftFusion 使用 NVSHMEM 这类 one-sided communication：

```text
push:
  本 GPU 主动把数据写到远端 GPU buffer

pull:
  本 GPU 主动从远端 GPU buffer 读取数据
```

这样同步责任转移给程序员，系统可以更细粒度地安排通信和计算。SwiftFusion 的实现目标是：

```text
只在 layer 开始和结束做必要的全局同步，
中间尽量用异步 push / pull 与 attention computation overlap。
```

## 6. 关键公式与推导

### 6.1 Ring Attention 的通信量

Ring Attention 中，每张 GPU 在每一步把自己的 `K/V` 发送给邻居，并从另一个邻居接收 `K/V`。对于 `P` 张 GPU，共有 `P` 步，最后一步不需要继续传给下一个。

论文给出的每 GPU 通信量是：

```text
V_ring = 2(P - 1)BLHD / P
       ≈ 2BLHD
```

### 推导

每张 GPU 本地的 `K/V` sequence 长度是 `L / P`。`K` 和 `V` 两个 tensor 合计通信量：

```text
per step = 2 * B * (L / P) * H * D
         = 2BLHD / P
```

Ring 需要传递 `P - 1` 次：

```text
V_ring = (P - 1) * 2BLHD / P
       = 2(P - 1)BLHD / P
```

当 `P` 增大时：

```text
2(P - 1) / P -> 2
```

所以：

```text
V_ring ≈ 2BLHD
```

### 直觉

Ring 的好处是流水线简单，可以边收 `KV` 边算 attention。但坏处是每张 GPU 最终几乎还是要接收完整一份全局 `KV`。GPU 变多后，每一步传得更少，但步数也变多，两者抵消。

这就是为什么 Ring 放在跨机器网络上会限制扩展性。

### 6.2 Ulysses Attention 的通信量

Ulysses 先用 all-to-all 把 full sequence 聚合到每张 GPU，同时把 head dimension 切分出去；attention 本身在本地完成；最后再 all-to-all 把输出恢复成 sequence-sharded layout。

论文给出的每 GPU 通信量是：

```text
V_ulysses = 4(P - 1)BLHD / P^2
          ≈ 4BLHD / P
```

### 推导

Ulysses 需要四次 all-to-all：

```text
Q: all-to-all
K: all-to-all
V: all-to-all
O: all-to-all
```

每次 all-to-all 中，每张 GPU 保留 `1/P` 的数据在本地，发送剩余 `(P - 1) / P`。原本每张 GPU 的 tensor 大小是：

```text
BLHD / P
```

一次 all-to-all 的通信量是：

```text
(P - 1) / P * BLHD / P
  = (P - 1)BLHD / P^2
```

四次合计：

```text
V_ulysses = 4(P - 1)BLHD / P^2
```

当 `P` 增大时：

```text
V_ulysses ≈ 4BLHD / P
```

### 直觉

Ulysses 用 head parallelism 换低通信量。GPU 越多，每张 GPU 需要通信的数据越少。

代价是：

```text
1. all-to-all 默认不容易 overlap；
2. P 不能超过可切分的 attention heads 数；
3. all-to-all 对网络拓扑和实现质量更敏感。
```

### 6.3 SwiftFusion 的 process mesh

SwiftFusion 把 `N * M` 张 GPU 组织成二维 mesh：

```text
P_u x P_r
```

其中：

```text
P_u: Ulysses / Torus 方向的并行度
P_r: Ring 方向的并行度
```

在最简单的情况，如果 attention heads 数 `H` 能匹配机器数 `N`：

```text
P_u = N
P_r = M
```

也就是：

```text
跨机器方向用 Ulysses / Torus
每台机器内部用 Ring
```

当 `H` 不能直接等于 `N` 时，论文选择：

```text
P_u = gcd(NM, H)
P_r = NM / P_u
```

### 推导

Ulysses 的 head sharding 要求 `H` 能被 `P_u` 整除。因此 `P_u` 不能随便取，它必须是 `NM` 和 `H` 的公因子。

要尽量减少跨机器通信，应尽可能增大 Ulysses 并行度：

```text
P_u 越大
  -> Ulysses 通信量越小
  -> 跨机器通信越少
```

所以取最大可行公因子：

```text
P_u = gcd(NM, H)
```

剩下的 GPU 维度交给 Ring：

```text
P_r = NM / P_u
```

### 直觉

这个公式背后的原则是：

```text
在 head dimension 允许的范围内，
尽量把并行度分给 Ulysses，
因为 Ulysses 是减少跨机器通信量的主要工具。
```

### 6.4 Torus Attention 的三阶段调度

Torus Attention 把 Ulysses 的四个 all-to-all 重新调度成三类阶段：

```text
Pull Q
Pull KV
Push O
```

其中 `T_{l,h}` 表示 tensor `T` 中 sequence partition 为 `l`、head partition 为 `h` 的 slice。

简化后的执行模型是：

```text
Pull Q:
  逐步拉取远端 Q chunk
  同时用已经可用的 Q 与本地 K/V 计算 attention

Pull KV:
  逐步拉取远端 K/V chunk
  同时用已收到的 K/V 更新 attention output

Push O:
  把已经算好的 O chunk 推送回对应 GPU
  同时计算本地剩余 O chunk
```

### 直觉

`Q` 和 `K/V` 的生命周期不同：

```text
Q chunk:
  决定哪些 query rows 要计算

K/V chunk:
  决定这些 query rows 要看哪些 key/value blocks

O chunk:
  是最终要恢复到原 sequence-sharded layout 的结果
```

如果简单地把四个 all-to-all 都先做完，再算 attention，就浪费了大量等待时间。Torus Attention 的关键是把“可计算的局部块”提前算掉，让慢链路通信在后台推进。

### 6.5 分布式 attention output 的合并

Ring / Torus 都会让同一个 `Q` 分别和不同 `KV` partitions 计算局部 attention output。要得到完整 attention，不能直接平均这些局部结果，而要像 FlashAttention 一样用 running max 和 running sum 稳定合并。

对第 `i` 个 `KV` partition，定义：

```text
m_i = rowmax(Q K_i^T)
l_i = rowsum(exp(Q K_i^T - m_i))
O_i = exp(Q K_i^T - m_i) V_i / l_i
```

把局部结果记成：

```text
A_i = [O_i, l_i, m_i]
```

合并两个局部结果 `A_i` 和 `A_j`：

```text
m = max(m_i, m_j)
l = l_i * exp(m_i - m) + l_j * exp(m_j - m)
O = (O_i * l_i * exp(m_i - m) + O_j * l_j * exp(m_j - m)) / l
```

### 推导

完整 softmax attention 是：

```text
softmax(QK^T)V
```

当 `K/V` 被切成多个 partition 时，分母需要覆盖所有 partitions：

```text
sum over all exp(score)
```

每个 partition 本地有自己的最大值 `m_i`。为了数值稳定，合并时要切换到共同最大值：

```text
m = max(m_i, m_j)
```

原来以 `m_i` 为基准的指数项，要换成以 `m` 为基准：

```text
exp(score - m)
  = exp(score - m_i) * exp(m_i - m)
```

所以局部 rowsum 也要乘上修正项：

```text
l_i -> l_i * exp(m_i - m)
l_j -> l_j * exp(m_j - m)
```

输出 `O_i` 原本已经除过 `l_i`，要先还原成加权分子，再合并：

```text
O_i * l_i * exp(m_i - m)
O_j * l_j * exp(m_j - m)
```

最后除以新的全局 `l`。

### 直觉

这套合并公式保证了：

```text
分块算 attention
  与
一次性在完整 K/V 上算 attention

得到同一个数学结果。
```

这对 SwiftFusion 很重要，因为它是系统加速，不是近似模型。它必须保持 attention 结果正确，不能靠牺牲生成质量换速度。

## 7. 推理执行过程

SwiftFusion 的 attention layer 可以写成一个可执行心智模型：

```text
input:
    local sequence-sharded Q, K, V
    GPU topology: N machines, M GPUs per machine
    attention heads H

setup:
    choose P_u = gcd(NM, H)
    choose P_r = NM / P_u
    build Torus / Ulysses / Ring process groups

for each attention layer:
    rearrange Q/K/V layout for process groups

    perform initial ScatterPush for local Ulysses-style chunks
    BarrierAll to ensure initial remote buffers are visible

    issue GatherPull operations for future Q/K/V chunks

    Pull Q stages:
        compute attention with currently available Q and local K/V
        overlap Q communication with computation

    Pull KV stages:
        wait for needed K/V chunk
        run RingAttn inside machine / ring group
        merge partial outputs using running max and sum

    Push O stage:
        ScatterPush completed output chunks to destination GPUs
        compute local remaining output while communication proceeds

    BarrierAll to ensure O is complete
    return restored sequence-sharded O
```

训练与推理的区别在这里不重要，因为 SwiftFusion 不是训练方法。它是推理执行引擎，目标是在不改变 DiT 模型权重和采样算法语义的前提下降低每个 sampling step 的 attention latency。

## 8. 实验结论

实验部分最好按“设计选择是否被证据支持”来读。

### 8.1 实验设置

硬件与软件：

```text
4 台 AWS p4de.24xlarge
每台 8 张 NVIDIA A100 40GB
机器内 NVSwitch
机器间 AWS EFA，最高 400 Gbps
CUDA 12.8.0
PyTorch 2.8.0
NCCL 2.27.3
NVSHMEM 3.4.5
```

工作负载：

```text
Flux 12B:
  image generation
  3072 x 3072 / 4096 x 4096
  24 heads, head dim 128

CogVideoX 5B:
  video generation
  20s / 40s, 768 x 1360
  24 heads, head dim 64
```

主要基线：

```text
USP:
  prior state-of-the-art sequence parallelism

TAS:
  SwiftFusion variant
  只使用 topology-aware scheduling
  不使用 Torus Attention 和 one-sided implementation

SFU:
  完整 SwiftFusion
```

### 8.2 整体性能证明了什么

论文报告 SwiftFusion 相对已有 SOTA 平均提升约 `1.35x`，最高 `1.77x`。

更细的结论是：

```text
2 台机器时:
  TAS 可能比 USP 差
  因为 Ulysses 与 Ring 的跨机通信量相近，
  但 TAS 的 all-to-all 还没有被 overlap。

超过 2 台机器时:
  TAS 平均可比 USP 快约 1.27x，最高 1.64x。
  这说明 topology-aware scheduling 本身能减少跨机通信量。

加入 Torus Attention 和 one-sided communication 后:
  SFU 进一步优于 TAS。
  这说明光减少通信量还不够，
  还必须隐藏剩余 all-to-all 延迟并减少同步开销。
```

这个实验支持论文的主张：

```text
SwiftFusion 的收益不是来自单一 trick，
而是 communication volume、overlap、synchronization 三者一起优化。
```

### 8.3 不同分布式配置证明了什么

论文在多种 `U x R y` 配置下比较：

```text
U x R y:
  Ulysses degree = x
  Ring degree = y
```

观察结果：

```text
TAS 平均比 USP 快约 1.47x，最高 2.54x。
SFU 平均比 USP 快约 1.61x，最高 3.11x。
```

这证明：

```text
把 Ring 尽量限制在机器内，
确实能显著降低跨机器通信瓶颈。
```

但论文也观察到，Ulysses degree 不是越大越无脑好。因为当 `R` 变小后，attention operator 被切得更碎，kernel launch overhead、warp 调度 bubble 和 GPU 利用率都会影响最终性能。

所以 SwiftFusion 的系统设计不是“最大化 Ulysses”这么简单，而是：

```text
在降低跨机通信量和保持 GPU compute efficiency 之间找平衡。
```

### 8.4 layerwise micro-benchmark 证明了什么

论文改变 head dimension、sequence length 和 batch size 来看单层 attention latency。

head dimension：

```text
D = 32:  平均约 1.12x speedup
D = 64:  平均约 1.28x speedup
D = 128: 平均约 1.32x speedup
```

这说明 head dimension 越大，attention computation 越容易填满 GPU，通信 overlap 的空间也更好利用。

sequence length：

```text
96k:  平均约 1.59x
128k: 平均约 1.27x
160k: 平均约 1.07x
192k: 平均约 1.14x
```

一个反直觉点是：sequence length 变长后，SwiftFusion 的相对 speedup 不一定更大。原因是 attention 计算量随 sequence length 近似二次增长，而通信量近似线性增长；当 layer 更 compute-bound 时，减少通信的相对收益会下降。

batch size：

```text
不同 batch size 下 SwiftFusion 都保持正收益，
但没有非常单调的趋势。
```

这说明它的收益不是只绑定某个 batch 设置，但具体加速比仍受 kernel utilization 和通信隐藏程度影响。

### 8.5 ablation 证明了什么

Appendix B 的消融实验把 SwiftFusion 的三件事逐步加上：

```text
USP
  -> + topology-aware scheduling = TAS
  -> + Torus Attention
  -> + one-sided communication = SFU
```

关键结论：

```text
TAS:
  平均 1.27x，最高 1.64x。
  证明跨机通信量减少本身有效。

Torus Attention:
  对 video workload 更重要。
  因为长视频 sequence 更长，Ulysses all-to-all 通信更容易成为瓶颈。

one-sided communication:
  对 image workload 也有明显价值。
  因为图像 sequence 相对短，Torus 本身能隐藏的通信有限，
  但减少同步和通信 kernel 开销仍能带来收益。
```

所以实验的因果链是：

```text
topology-aware scheduling:
  主要减少慢链路通信量

Torus Attention:
  主要隐藏跨机 all-to-all latency

one-sided communication:
  主要减少同步开销和通信实现开销
```

## 9. 本次疑问记录

### SwiftFusion 会改变生成质量吗？

理论上不会。它优化的是 DiT attention layer 的分布式执行方式，不改变模型权重、扩散采样步骤或 attention 的数学结果。

关键前提是：

```text
分布式 attention output 必须和单机完整 attention output 数学等价。
```

这也是 Appendix C 合并公式重要的原因。

### 为什么慢的机器间网络反而用 all-to-all？

因为“通信 pattern 简单”不等于“通信代价低”。Ring 的 pattern 简单、容易 overlap，但跨机通信量不随机器数下降。Ulysses 的 all-to-all 更复杂，但通信量会随并行度下降。

SwiftFusion 的取舍是：

```text
跨机器:
  宁可处理复杂 all-to-all，
  也要先降低总通信量。

机器内:
  网络足够快，
  可以承受 Ring 的较高通信量。
```

### Torus Attention 和 Ulysses 的关系是什么？

Torus Attention 可以理解为 Ulysses Attention 的通信-计算重排版本。它不改变 Ulysses 的最终数据布局目标，而是把 all-to-all 拆成 chunks，并按 attention 计算依赖重新排序。

```text
Ulysses:
  all-to-all -> local attention -> all-to-all

Torus:
  chunk communication overlaps chunk attention
```

### 为什么需要自定义 attention kernel？

Torus / Ring 会让一个 attention output 来自多个 `Q` 和多个 `KV` chunks。如果每个 chunk 单独调用 FlashAttention，再额外 launch kernel 做 merge，会引入 kernel launch overhead 和全局内存访问。

论文因此实现了能处理多个 `Q/KV` tensors 并融合 merge 操作的 CUDA kernel，目标是：

```text
保持 FlashAttention 风格的数值稳定合并，
同时避免把分块调度的收益浪费在 kernel overhead 上。
```

### 这篇论文为什么放在 diffusion 目录下？

它不是提出新的 diffusion objective，而是服务于 Diffusion Transformers 的分布式推理。这里的 diffusion 重点是系统负载形态：

```text
DiT sampling step 多
activation 大
attention sequence 长
高分辨率图像 / 长视频推理需要多卡并行
```

## 10. 最小心智模型

如果只能记住一句话：

```text
SwiftFusion = topology-aware sequence parallelism + Torus Attention + one-sided communication for DiT inference.
```

更完整一点：

```text
DiT 长序列推理:
  activation 太大，必须 sequence parallel

sequence parallel:
  省显存，但 attention 需要跨 GPU 通信

Ring:
  可 overlap，但通信量高

Ulysses:
  通信量低，但 all-to-all 默认不可 overlap

USP:
  把 Ring 放在跨机器路径上，跨机通信量仍高

SwiftFusion:
  inter-machine 用 Ulysses / Torus 降低并隐藏慢通信
  intra-machine 用 Ring 利用快互联
  one-sided communication 减少同步和 kernel 开销
```

最短逻辑链是：

```text
高分辨率 / 长视频 DiT 让 attention sequence 变长
sequence parallelism 必须通信全局 KV
跨机器网络比机器内网络慢
Ring 的跨机通信量不随机器数下降
Ulysses 能降低通信量但不能天然 overlap
Torus 把 all-to-all 拆块并与 attention 计算流水化
one-sided communication 减少隐式同步
因此 SwiftFusion 降低分布式 DiT 推理延迟
```

## 11. 盲点与后续阅读

最重要的盲点是：

```text
SwiftFusion 的收益依赖硬件拓扑、attention heads 可切分性、
通信库质量和 workload 是否 communication-bound。
```

几个后续需要特别验证的问题：

1. 当 attention heads 数 `H` 无法给 Ulysses 足够并行度时，SwiftFusion 的优势会下降多少。
2. 在 H100 / B200、NVLink Switch、InfiniBand NDR 或不同云厂商网络上，Torus 与 one-sided communication 的收益是否保持稳定。
3. 当 sequence length 极长导致 attention 更 compute-bound 时，通信优化的边际收益会不会进一步下降。
4. 自定义 CUDA kernel 的维护成本和跨 GPU 架构移植成本是否会抵消一部分工程收益。
5. SwiftFusion 是否能与 PipeFusion、DistriFusion、ScaleFusion 这类跨 sampling step 或时空结构优化叠加，还是会在通信调度上互相冲突。
6. 论文主要展示 one sampling step / attention layer latency，完整在线 serving 中还需要结合 scheduler、batching、VAE、I/O 和 diffusion sampler 开销重新评估端到端收益。

这篇论文对 diffusion system 的启发是：

```text
扩散模型的系统瓶颈不只在 denoising steps 多，
也在每一步 DiT attention 的 activation 和通信形态。

当模型进入长视频和超高分辨率阶段，
“如何跨设备移动 attention tensor”
会和“如何减少采样步数”一样成为核心加速问题。
```
