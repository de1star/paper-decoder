# DFlash: Block Diffusion for Flash Speculative Decoding 阅读笔记

论文：DFlash: Block Diffusion for Flash Speculative Decoding  
链接：https://arxiv.org/abs/2602.06036  
PDF：https://arxiv.org/pdf/2602.06036  
作者：Jian Chen, Yesheng Liang, Zhijian Liu  
常用简称：DFlash

## 1. 阅读计划回顾

这篇论文不要先读成“又一个 diffusion language model”。更好的入口是：

```text
LLM 解码慢
  -> 不是因为 target model 每次不能并行算 logits
  -> 而是因为新 token 必须一个接一个产生
  -> speculative decoding 把验证阶段并行化
  -> 但很多 draft model 仍然自回归生成候选 token
  -> 所以 DFlash 把 diffusion 放在 draft 阶段
  -> 用 target AR model 的验证保证最终输出 lossless
```

建议按以下顺序阅读：

1. `Abstract`、`Introduction`：确认论文的主张不是“扩散模型替代自回归模型”，而是“扩散模型适合作为并行 drafter”。
2. `Preliminaries`：先读速度公式，明确 speedup 由 `T_draft`、`T_verify` 和 `tau` 共同决定。
3. `Inference`：读 DFlash 如何从 target model 提取 hidden features，并通过 KV injection 条件化 draft model。
4. `Training`：读随机 anchor block、sparse attention mask、位置衰减 loss、共享 embedding/LM head。
5. `Experiments`：按“每个结果证明了哪个设计选择有效”来读，而不是只抄 speedup 数字。
6. `Ablation Study` 和 Appendix：回头验证没有 target context、不同 draft depth、hidden feature 数量、block size、loss decay 分别影响什么。

## 2. 阅读主线

显式问题是：

```text
如何突破自回归 LLM token-by-token decoding 的延迟瓶颈，
并在不改变目标模型输出分布的前提下提高推理速度。
```

真正的结构张力是：

```text
speculative decoding 已经让 target model 的验证阶段并行化，
但 draft 阶段常常还是 autoregressive。

如果 drafter 太小：
  draft 很快，但 token 质量差，target model 接受长度短。

如果 drafter 更强：
  acceptance length 变长，但 drafting latency 又上升。

diffusion LLM 能并行生成 token block，
但独立作为最终语言模型时质量通常弱于 autoregressive LLM。
```

第一原则是：

```text
在 speculative decoding 里，drafter 不需要独立生成最终答案。
它只需要提出一批足够容易被 target model 接受的候选 token。

最终正确性由 target autoregressive model 的并行验证保证。
```

所以 DFlash 的核心重构是：

```text
不要让 diffusion LLM 和 autoregressive LLM 竞争最终生成质量。
让 diffusion model 做并行草稿生成，
让 autoregressive target model 做质量校验。
```

## 3. 论文要解决的问题

自回归 LLM 的解码过程是：

```text
given context
  -> 预测第 1 个新 token
  -> 把第 1 个 token 加回 context
  -> 预测第 2 个新 token
  -> 继续循环
```

这个过程强依赖顺序。即使单次 forward 可以在 GPU 上高效计算，下一步的输入也必须等上一步 token 出来。

speculative decoding 的基本想法是：

```text
小模型先猜一串未来 token
  -> 大 target model 一次性并行验证这些 token
  -> 接受最长正确前缀
  -> 如果全部接受，再额外得到 target model 的 bonus token
```

它是 lossless 的：最终输出分布仍然由 target model 决定，draft model 只影响速度。

问题在于，许多强 speculative decoding 方法的 drafter 仍然是 autoregressive。它们虽然比 target model 小，但仍要一个 token 一个 token 地生成草稿。这带来两个上限：

1. `T_draft` 随 speculation budget 线性增长。
2. 小 drafter 容量有限，`tau` 很快饱和。

DFlash 认为这里存在一个更合适的位置给 diffusion model：

```text
diffusion model 不必完成最终语言建模
  -> 它只负责一次 forward 并行提出 block tokens
  -> target AR model 负责验证
  -> diffusion 的质量短板被 verification 吸收
  -> diffusion 的并行优势被 drafting 放大
```

## 4. 核心符号

```text
M_t: target model，也就是原本要加速的 autoregressive LLM。
M_d: draft model，也就是 DFlash 中的 lightweight block diffusion model。

gamma: speculative decoding 中每轮草稿 token 数，也可理解为 block size / speculation budget。
tau: 每轮平均接受 token 数，包含 target model 额外给出的 bonus token。

T_draft: 每轮生成 draft tokens 的时间。
T_verify: target model 并行验证 draft tokens 的时间。
L: speculative decoding 的平均每 token 延迟。
L_target: target model 原始 autoregressive decoding 的每 token 延迟。
eta: speedup，eta = L_target / L。

t_step: autoregressive drafter 生成一个 token 的单步延迟。
t_parallel: diffusion drafter 一次并行生成整个 block 的延迟。

block size: diffusion drafter 每次预测的 token block 长度。
anchor token: 训练中随机采样的干净 response token，用作一个 block 的起点。
mask token: block 内需要 draft model 并行预测的位置。

target hidden features: 从 target model 多个层抽取的 hidden states。
feature fusion: 把多层 hidden states 拼接后投影成紧凑 context feature。
KV injection: 将 fused context feature 注入 draft model 每一层的 Key/Value cache。
```

注意一个符号复用：

```text
论文中 gamma 既用于 speculative decoding 的 draft 长度，
也在 loss weighting 中作为指数衰减超参数。

读公式时要看上下文：
  速度分析里的 gamma: 每轮 draft token 数。
  loss 里的 gamma: 控制早位置权重衰减速度。
```

## 5. 方法框架

DFlash 的方法可以压缩成一句话：

```text
用 target AR model 的 hidden features 条件化一个小型 block diffusion drafter，
让它一次 forward 并行生成一段候选 token，
再由 target model 并行验证。
```

整体流程是：

```text
prompt / accepted tokens
  -> target model prefill 或 verification
  -> 抽取 target hidden features
  -> 融合多层 hidden features
  -> 注入 draft model 每一层的 KV cache
  -> block diffusion drafter 并行预测下一段 tokens
  -> target model 验证并接受最长前缀
  -> 进入下一轮
```

### 为什么要用 target hidden features

一个小 diffusion drafter 如果只看 token context，就必须自己推理未来 token。论文的 naive ablation 表明，这种小 diffusion drafter 的 speedup 通常只有 2-3 倍左右。

DFlash 的判断是：

```text
target model 的 hidden states 不只是当前 token 表示。
它们隐含了长程依赖、任务语义，以及多个未来 token 的预测信息。
```

因此 draft model 不应该从零开始猜，而应该借用 target model 已经建好的上下文表示。

### 为什么不是只把 feature 加到输入

EAGLE-3 也使用 target model 的 hidden features，但主要是把这些 feature 融入 draft model 的输入。DFlash 认为这样有一个问题：

```text
feature 只在输入层进入
  -> draft model 层数越深
  -> target feature 信号越容易被稀释
  -> 加深 drafter 对 acceptance length 的收益变弱
```

DFlash 改成 KV injection：

```text
fused target context feature
  -> 投影成每层可用的 Key / Value
  -> 存入 draft model KV cache
  -> 每一层都能持续访问 target context
```

这使得 draft model 加深时，acceptance length 能更有效地增长。

### 为什么 diffusion drafting 改变设计空间

autoregressive drafter 的成本近似是：

```text
draft 8 tokens  = 8 次顺序 forward
draft 16 tokens = 16 次顺序 forward
```

所以它为了低延迟必须很浅，通常没有足够容量预测高质量 token。

block diffusion drafter 的成本更接近：

```text
draft 16 tokens = 1 次并行 forward
```

这意味着它可以用更深的 draft layers 换更高的 draft quality，而不必像 autoregressive drafter 那样为每个 token 额外付一次顺序延迟。

## 6. 关键公式与推导

### 6.1 speculative decoding 的平均延迟

论文给出的平均每 token 延迟是：

```text
L = (T_draft + T_verify) / tau
```

其中：

```text
T_draft: 一轮中生成草稿 token 的时间。
T_verify: target model 一次性验证这些 token 的时间。
tau: 一轮中平均被接受的 token 数，包含 bonus token。
```

speedup 是：

```text
eta = L_target / L
```

### 推导

一轮 speculative decoding 的总耗时是：

```text
T_cycle = T_draft + T_verify
```

如果这一轮平均推进 `tau` 个 token，那么平均每个 token 分摊到的时间就是：

```text
L = T_cycle / tau
  = (T_draft + T_verify) / tau
```

和原始 target model 的单 token 延迟 `L_target` 相比，速度提升就是：

```text
eta = L_target / L
```

### 直觉

这个公式把 speculative decoding 的优化目标说得很清楚：

```text
要么让 draft 更便宜，降低 T_draft；
要么让 draft 更准，提高 tau；
要么降低验证开销 T_verify。
```

DFlash 的设计同时压低 `T_draft`、提高 `tau`：

```text
block diffusion parallel drafting
  -> 降低 T_draft

target feature conditioning + KV injection
  -> 提高 tau
```

### 6.2 autoregressive drafter 的 draft cost

autoregressive drafter 生成 `gamma` 个 token 时：

```text
T_draft = gamma * t_step
```

### 推导

自回归生成必须顺序执行：

```text
第 1 个 draft token: t_step
第 2 个 draft token: t_step
...
第 gamma 个 draft token: t_step
```

因此总 draft 时间是：

```text
T_draft = t_step + t_step + ... + t_step
        = gamma * t_step
```

### 直觉

当 `gamma` 变大时，autoregressive drafter 需要更多顺序步骤。即使每一步很小，总成本也会线性累积。

这就是为什么 EAGLE-3 这类方法常被迫使用很浅的 drafter：否则 draft 还没被 target model 验证，自己已经花掉太多时间。

### 6.3 diffusion drafter 的 draft cost

block diffusion drafter 生成一个 block 时：

```text
T_draft = t_parallel
```

### 推导

DFlash 不把 `gamma` 个 token 看成 `gamma` 次顺序预测，而是把它们看成一个 block 内的并行预测：

```text
masked block
  -> one forward pass
  -> predict all masked positions
```

因此 draft 时间主要由一次并行 forward 决定：

```text
T_draft = t_parallel
```

### 直觉

这个公式不是说 diffusion drafter 完全不受 block size 影响，而是说它不再像 autoregressive drafter 那样按 token 数线性增长。

因此 DFlash 能使用更深的 draft model，并且一次预测更多 token，在 draft quality 和 drafting cost 之间得到更好的 Pareto frontier。

### 6.4 位置衰减 loss

DFlash 对 block 内第 `k` 个 token 使用指数衰减权重：

```text
w_k = exp(-(k - 1) / gamma)
```

其中这里的 `gamma` 是 loss decay 的超参数。

### 推导

当 `k = 1` 时：

```text
w_1 = exp(0) = 1
```

第一个预测 token 权重最大。

当 `k` 变大：

```text
k - 1 变大
-(k - 1) / gamma 变得更负
exp(...) 变小
```

所以越靠后的 token 权重越低。

### 直觉

在 speculative decoding 里，draft block 的接受是前缀式的：

```text
第 1 个 token 错了
  -> 后面 token 即使猜对也不会被接受

第 1 到第 4 个 token 都对，第 5 个错
  -> 只能接受前 4 个
```

因此早位置 token 的错误成本更高。DFlash 让 loss 更重视 block 前部，是为了更快提高 acceptance length，而不是平均地优化所有位置。

## 7. 训练过程

DFlash 训练的目标不是训练一个完整独立的 diffusion LLM，而是训练一个和冻结 target model 对齐的 draft adapter。

训练流程可以写成：

```text
for each prompt + target-generated response:
    run frozen target model on the clean sequence
    extract hidden features from selected layers
    fuse hidden features into target context
    randomly sample anchor tokens in the response
    build masked blocks starting from anchors
    inject target context into draft model KV cache
    predict masked tokens in parallel
    compute weighted cross-entropy loss
    update only draft transformer layers
```

### 7.1 用 target-generated responses 训练

论文使用约 800K 样本，来源包括 NVIDIA Nemotron Post-Training Dataset V2 和 CodeAlpaca。关键不是直接使用原始 answer，而是构造 target model 自己生成的 responses。

这样做的原因是：

```text
speculative decoding 最终由 target model 验证。
draft model 越贴近 target model 的输出分布，
越容易获得更长 acceptance length。
```

### 7.2 随机 anchor block

标准 block diffusion training 通常把 response 均匀分块，然后随机 mask block 内位置。

DFlash 改成：

```text
在 response 中随机采样 anchor token
  -> anchor token 作为 block 第一个位置
  -> mask 后续 positions
  -> 训练 draft model 并行预测 block_size - 1 个未来 token
```

这更贴近 inference：

```text
每轮 speculative decoding 开始时，
draft model 都会条件化在一个已经由 target model 确认的 clean token 上。
```

随机 anchor 还扩大了训练覆盖面，让模型见到更多不同位置的 target context features。

### 7.3 sparse attention mask

训练时，多个 masked blocks 会拼在一个 sequence 中一起处理。为了防止信息泄露，DFlash 使用 sparse attention mask：

```text
block 内:
  可以 bidirectional attention
  可以看对应 target context features

block 间:
  不允许互相 attention
```

这让训练能在单次 forward/backward 中处理多个 block，同时保持和 inference 一致的因果约束。

### 7.4 shared embedding and LM head

DFlash draft model 共享 target model 的 token embedding 和 LM head，并冻结它们。只训练中间的 draft transformer layers。

这样做有三个效果：

```text
减少 trainable parameters
让 draft model 留在 target model 的 token 表示空间里
强化“draft adapter”而不是“独立语言模型”的定位
```

## 8. 推理 / 解码过程

DFlash 的 inference 可以写成：

```text
input prompt
target model prefill
generate first token / obtain accepted prefix

repeat:
    extract selected target hidden features
    fuse features through a lightweight projection
    inject fused features into each draft layer's KV cache
    draft model predicts a token block in one parallel forward
    target model verifies the proposed block in parallel
    accept the longest valid prefix
    add target model bonus token when applicable
until stopping condition
```

训练和推理的关键差异是：

```text
训练时:
  target-generated clean response 已知。
  可以随机采样很多 anchor blocks。
  目标是让 draft model 学会预测 target model 会接受的未来 token。

推理时:
  未来 token 未知。
  draft model 必须根据 accepted prefix 和 target context features 提出候选 block。
  target model 决定哪些 token 被接受。
```

DFlash 的质量保证来自最后一步：

```text
draft model 只提案
target model 做验证
```

所以只要 speculative decoding 的验证规则保持正确，最终输出仍是 target model 的 lossless acceleration。

## 9. 实验结论

实验部分最好按“证明链”来读。

### 9.1 Qwen3 instruct models

在 Qwen3-4B 和 Qwen3-8B、thinking mode disabled 的设置下，DFlash 与 EAGLE-3 对比：

```text
temperature = 0:
  DFlash 平均约 4.9x speedup
  EAGLE-3 tree size 16 平均约 1.8x
  EAGLE-3 tree size 60 平均约 2.0x

temperature = 1:
  DFlash 仍保持约 4.0x 以上平均 speedup
  sampling 下优势没有消失
```

这证明：

```text
parallel diffusion drafting 不只是 greedy decoding 的特殊技巧。
在 sampling 设置下，DFlash 仍能保持较高 acceptance length。
```

### 9.2 reasoning models

thinking mode enabled 时，response 通常更长，推理成本更高。DFlash 在 Qwen3 reasoning benchmarks 上仍能取得约 4x 左右 speedup。

这证明：

```text
DFlash 对长推理链有实际价值。
因为越长的 generation，越能放大每轮 decoding 加速的收益。
```

### 9.3 SGLang production serving

论文在 SGLang 上测试 Qwen3-4B、Qwen3-8B 和 Qwen3-Coder-30B-A3B-Instruct，覆盖 concurrency 1 到 32。

重要观察是：

```text
低并发时 speedup 更高；
并发升高后 speedup 下降，但仍然保持正收益。
```

这说明 DFlash 的优势不是只存在于离线 benchmark，也能进入 serving 框架。但它也提醒我们：

```text
真实服务中，verification cost、batch size、GPU compute/memory 状态会改变最优 block size。
```

### 9.4 LLaMA-3.1 对比

论文还在 LLaMA-3.1-8B-Instruct 上与官方 EAGLE-3 checkpoint 对比。在相同训练数据下，DFlash 在 GSM8K、HumanEval、Alpaca 等任务和多个并发设置下保持更稳定 speedup。

这证明：

```text
DFlash 不是只对 Qwen3 特化。
它的核心机制可以迁移到不同 target AR models。
```

### 9.5 ablation：没有 target context 不够

5-layer block diffusion drafter 如果不接入 target context features，speedup 通常只有 2-3x。

这证明：

```text
DFlash 的关键不是“用 diffusion”四个字。
关键是“用 target model hidden features 条件化 diffusion drafter”。
```

### 9.6 ablation：draft depth 有最优点

更深的 draft model 通常有更长 acceptance length，但也带来更高 drafting cost。论文中 5-layer draft model 在平均 speedup 上优于 8-layer，因为它更好地平衡了质量和延迟。

这证明：

```text
acceptance length 不是唯一目标。
真正目标是 (T_draft + T_verify) / tau 最小。
```

### 9.7 ablation：hidden feature 数量

从更多 target layers 提取 hidden features 可以提升 acceptance length 和 speedup，但代价是训练成本和 hidden state 缓存成本增加。

这证明：

```text
target features 确实提供有效未来信息，
但不是免费信息。
```

### 9.8 ablation：block size

block size 16 的模型在 math/code tasks 上通常能获得更高 acceptance length。训练时用大 block size 的模型可以较好泛化到较小 inference block size，但反过来不成立。

这说明：

```text
训练大 block 是更稳的选择；
推理时可以根据服务负载动态缩小 block size。
```

论文也把 adaptive block-size scheduling 留给未来工作。

### 9.9 ablation：loss decay

位置衰减 loss 让训练更快、更好收敛。

这证明：

```text
speculative decoding 的训练目标应该匹配 prefix acceptance 机制。
早位置 token 更重要，所以 loss 也应该更重视早位置。
```

## 10. 本次疑问记录

### 为什么 diffusion model 质量弱还能加速高质量 LLM？

因为 DFlash 不让 diffusion model 负责最终输出质量。它只负责提出候选 token，最终接受与否由 target model 决定。

```text
diffusion drafter 影响速度
target AR model 决定输出
```

### 为什么 target hidden features 比 logits 更有用？

logits 是对下一个 token 的最终分布投影，信息已经被压到词表维度。hidden features 保留了更多中间语义、长程依赖和潜在未来 token 信息。

DFlash 借用这些 hidden features，是为了让小 drafter 不必从 token context 中重新推理。

### 为什么要 KV injection，而不是普通 feature concatenation？

普通输入融合只在第一层注入信息，随着 draft layers 加深，target feature 信号会被逐渐稀释。

KV injection 把 target context 放入每一层的注意力上下文，让 draft model 每一层都能持续访问它。

### DFlash 还是 diffusion model 吗？

是，但它不是传统图像扩散那种多步逐渐去噪的生成器。这里更接近 block diffusion language modeling：在一个 token block 内并行预测 masked positions，并且论文为了速度使用单次 forward drafting。

这也是 DFlash 的定位：

```text
不是完整 dLLM
而是 diffusion-style block drafter
```

### 为什么说它是 lossless acceleration？

speculative decoding 的校验步骤保证最终接受的 token 与 target model 的生成规则一致。draft model 如果猜错，只会降低接受长度和速度，不会直接改变最终输出分布。

## 11. 最小心智模型

如果只能记住一个模型，可以记成：

```text
DFlash = target-aware block diffusion drafter for speculative decoding
```

更完整一点：

```text
target AR model:
  负责正确性、最终分布、verification

block diffusion drafter:
  负责一次 forward 并行提出多个 future tokens

target hidden features:
  把 target model 的未来预测能力借给小 drafter

KV injection:
  让这种条件信息在 draft model 每一层都不被稀释

loss decay + random anchor blocks:
  让训练目标贴近 inference-time prefix acceptance
```

所以 DFlash 成立的最短逻辑链是：

```text
speculative decoding 的速度 = draft 成本低 + 接受长度高
autoregressive drafting 成本随 token 数线性增长
diffusion drafting 可以并行生成 block
小 diffusion drafter 独立预测不够准
target hidden features 提供高质量上下文
KV injection 让上下文持续影响每层
因此 DFlash 同时降低 T_draft 并提高 tau
```

## 12. 盲点与后续阅读

最重要的盲点是：

```text
DFlash 把 diffusion drafter 的质量问题转化成 target-conditioned adapter 训练问题。
这降低了推理时 draft latency，
但引入了隐藏状态抽取、缓存、训练数据构造和服务框架集成成本。
```

几个后续需要特别验证的问题：

1. target hidden features 的存储和预计算成本在更大模型、更长 context 下是否会成为训练瓶颈。
2. block size 的最优值是否随 batch size、并发数、verification backend 和任务类型显著变化。
3. KV injection 对不同架构的 target model 是否同样稳定，尤其是 MoE、长上下文模型和多模态模型。
4. DFlash 的 lossless 保证依赖 speculative decoding verification；如果实际工程实现为了吞吐做近似，需要重新检查输出分布是否仍然严格一致。
5. diffusion drafter 在高 temperature、开放式长文本、工具调用或强格式约束任务中，acceptance length 是否会更不稳定。

这篇论文对 diffusion LLM 的启发是：

```text
扩散模型未必一定要和 autoregressive model 竞争最终生成器位置。
在某些系统结构里，它可以成为并行化模块，
专门解决 autoregressive pipeline 中最顺序、最昂贵的一段。
```
